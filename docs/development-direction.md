# InkFlow 后续开发方向

更新日期：2026-07-22

## 产品方向

InkFlow 已具备 Hexo + GitHub Pages 写作客户端的完整主流程：文章同步、
Markdown 编辑、元数据、图床、仓库草稿/发布、文件树、友链和日志。

下一阶段不以继续堆叠零散功能为目标，而是将产品定位为：

> **面向个人博客作者的、安全可靠的 GitHub + Hexo 写作工作台。**

所有新功能都应优先满足以下原则：

1. 不因网络、权限或同步异常误改用户本地数据。
2. 远端变更、冲突和失败对用户可见且可恢复。
3. 发布前的质量检查可复现，不能依赖人工记忆。
4. 在稳定基础上再提升写作效率和首次使用体验。

## 近期路线

| 阶段 | 主题 | 主要产出 |
| --- | --- | --- |
| P0 | 可靠性版本 | 测试与 CI 门禁、同步安全、原子移动发布、frontmatter 数据保真 |
| P1 | 写作与冲突体验 | 冲突对比/处理、自动保存与历史、文章大纲、查找替换、发布前检查 |
| P2 | 上手与扩展 | 配置向导、多博客 Profile、按真实需求扩展对象存储/图床 |

## P0：可靠性版本

### 目标

让每次提交和发版都经过可重复的检查，并避免同步或发布异常造成文章
状态错误、文件重复或 frontmatter 语义变化。

### 工作项

#### 1. 测试与质量门禁

- 修复测试代码与当前 API 的漂移，测试当前行为而非恢复已废弃接口。
- 为关键逻辑补充单元测试：同步目录状态、精确的 post/draft 删除标记、
  原子移动发布、frontmatter 类型保留。
- Pull Request 和主分支提交均执行：
  `dart format --set-exit-if-changed`、`flutter analyze`、`flutter test`。
- Release 工作流在构建各平台产物前复用同一质量检查。
- 将 vendored 的 sqlite3 hook 补丁排除在应用静态分析范围外；它不是应用
  运行时代码，也不应通过增加不必要的直接依赖来满足分析器。

**验收标准**

- `flutter analyze` 无 error/warning。
- `flutter test` 成功执行全部测试。
- 不通过质量检查时，PR 和标签发布均不能进入构建阶段。

#### 2. 同步安全

- GitHub 目录读取结果明确区分：成功、目录不存在、请求失败。
- 只有文章目录都完整读取成功时，才根据远端文件集合标记本地文章为
  `remoteDeleted`。
- 目录不存在时允许空仓库继续使用，但跳过删除对账并写入日志。
- 增量同步的 commit 列表请求失败必须显式失败/降级，不能把失败当作“没有
  新提交”。
- 根据完整远端路径标记删除，避免同名 post 与 repo draft 被同时误标。

**验收标准**

- 401、403、5xx、网络错误不会改变任何本地文章的远端状态。
- `source/_posts/a.md` 被删除时，不会影响
  `source/_drafts/a.md`。
- 空的全新仓库可以同步成功，但不会执行删除对账。

#### 3. 发布安全

- 仓库草稿与正式文章之间切换时，通过 GitHub Git Data API 在一个 commit
  中写入新文件并删除旧文件。
- 并发更新导致分支引用变化时，操作安全失败，不覆盖远端数据。
- 新目标路径已存在时，拒绝覆盖并提示用户处理冲突。

**验收标准**

- 草稿发布成功后，远端只保留正式文章；反向操作同理。
- 任一请求失败时，不会出现“本地显示成功、远端只完成一半”的状态。
- 切换目标路径冲突时不覆盖已有远端文章。

#### 4. Frontmatter 数据保真

- 自定义字段在内存和数据库中保留 YAML 原始类型，而非一律转为字符串。
- 未编辑的 boolean、数字、列表和对象字段在打开、保存、同步后保持语义。
- 元数据页编辑字段时，仅把实际编辑过的字段转为用户输入的字符串。

**验收标准**

- `comments: true` 不会被写成 `comments: "true"`。
- 列表、数字和嵌套对象字段在未编辑时可 round-trip。

### 非目标

- **不做旧用户数据或凭据迁移。** 当前仅有一个使用者，若需重置可直接
  清除本地应用数据并重新配置。
- 不在 P0 引入新的图床、主题、音乐或社交功能。
- 不进行一次性项目级重构；新增逻辑只做必要的、小范围可测试抽取。

## P1：写作与冲突体验

P0 稳定后，优先实现：

1. 远端 SHA 冲突时的差异查看，以及“保留本地 / 使用远端 / 手工合并”。
2. 自动保存、本地历史版本与异常退出恢复。
3. 文章大纲、字数/预计阅读时长、查找替换、常用快捷键和发布前检查。

## P2：上手与扩展

1. 首次配置向导：验证 Token、仓库、分支、Hexo 目录及 GitHub Pages。
2. 多博客 Profile：隔离仓库、分支、图床和本地缓存。
3. 依据实际使用反馈扩展 S3、R2、OSS/COS 等存储，而非预先堆叠提供商。

## 持续工程约束

- 编辑器、设置和首页已较大；新业务逻辑优先放入可独立测试的服务或纯
  Dart 逻辑中。
- 每项路线图工作必须同时包含验收测试与用户可见的失败反馈。
- 文档中的“已完成”仅在 CI 质量门禁通过后勾选。
